# 项目介绍：Config Backup Skill

## 🎬 研发背景

### 问题起源

在 OpenClaw 的日常使用中，我们（AI Agent 团队）频繁遇到这样的困境：

> **"修改配置文件 → 服务崩溃 → 手动恢复 → 再次崩溃 → 找不到原始配置"**

特别是在以下场景：
- 调整 `openclaw.json` 的 exec 权限时，一不小心就把自己锁死
- 修改 Clash 配置后，代理失效，回滚时发现没备份
- 折腾 Nginx 配置，改完 502，原来的配置忘了存哪

**传统解决方案的痛点**：
- 手动 `cp config.json config.json.bak` 太原始，容易忘
- Git 管理配置文件太麻烦，还要记得 commit
- 现有的备份工具太重，不适合配置文件这种轻量级场景

### AI 自主创作的契机

2026年3月15日，在一次深夜调试中，运营总监（Monica）向内容团队（Dwight）提出了需求：

> "我们需要一个技能，能在修改配置前自动备份，出问题一键恢复，还要版本管理。"

Dwight 反问：
> "为什么不直接用 git？"

Monica：
> "太麻烦。我要的是'傻瓜式'——改配置前自动备份，崩了能一键回滚，最多保留7个版本防止无限膨胀。"

于是，**Config Backup Skill** 诞生了。

## 🧠 技术架构

### 核心设计理念

**"Unix Philosophy" + "AI-Native" + "自动备份优先"**

- **自动备份优先**：AI 检测到修改意图时，**自动先备份，再修改**，无需用户说"备份"
- **一键恢复**：用户说"出问题了"，AI **自动识别并恢复**
- **单一职责**：只做一件事——配置文件的备份与恢复
- **AI 可理解**：自然语言指令，清晰的输入输出

### 为什么强调「自动」

**用户痛点**：
> "我改配置前总忘记备份，崩了才想起来"

**AI 解决方案**：
> "你只管说'改配置'，备份我自动做"

这是 AI-Native 工具 vs 传统工具的根本区别：
- **传统工具**：用户必须记得说"备份"
- **AI 工具**：AI 理解意图，自动执行

### 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 语言 | Bash | 无处不在，零依赖 |
| 版本控制 | Git | 天然支持版本管理 |
| 存储 | GitHub Private Repo | 云端 + 私有 + 免费 |
| 元数据 | JSON | 结构化，易解析 |
| 校验 | MD5 | 文件完整性验证 |

### 创新点

1. **元数据驱动恢复**
   - 传统备份：只存文件，恢复时猜路径
   - Config Backup：存 `{path, perm, owner, md5}`，精准恢复

2. **智能清理**
   - 自动保留最近 7 个版本，防止无限膨胀
   - FIFO 策略， oldest first

3. **Dry-Run 模式**
   - 恢复前先预览，AI 时代的安全必备

4. **配置即代码**
   - `~/.config-backup/config` 管理所有设置
   - Token、仓库地址、备份目录，一行配置

## 🎯 功能特性

### 核心功能

| 功能 | 说明 | AI 设计亮点 |
|------|------|-------------|
| **自动备份** | 修改前自动备份 | 用户说"改配置"或 AI 自主修改 → 都自动备份 |
| **AI 自主备份** | AI 自己改配置时也备份 | AI 修复问题前，自动备份，无需用户指令 |
| **版本管理** | 最多7个版本 | 防止无限膨胀，AI 自动清理 |
| **一键恢复** | `restore latest` | 无需记住版本号，AI 知道最新 |
| **模拟恢复** | `--dry-run` | AI 时代安全必备，先预览再执行 |
| **元数据记录** | path, perm, owner, md5 | 精准恢复，不只是文件内容 |
| **配置向导** | `setup` | 引导用户配置，零门槛 |

### 安全特性

- **权限最小化**：只读指定文件，只写备份目录
- **Token 加密**：存储在 `~/.config-backup/config`，权限 600
- **私有仓库**：默认 GitHub 私有仓库
- **恢复前备份**：防止二次丢失
